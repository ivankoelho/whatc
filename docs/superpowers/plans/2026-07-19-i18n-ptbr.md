# pt-BR Internationalization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make WhatC fully available in Brazilian Portuguese with exactly two selectable UI languages — English (`en`) and Português (Brasil) (`pt-BR`) — with pt-BR as the default.

**Architecture:** Preserve the existing `vue-i18n` + auto-discovery setup. Add a `pt-BR.json` locale, delete the four extra locales, adjust default/detection logic in `i18n/index.ts`, and extract ~200 remaining hardcoded UI strings into i18n keys. Two Node validation scripts (key parity + hardcoded scan) act as the objective test gates.

**Tech Stack:** Vue 3, TypeScript, vue-i18n, shadcn-vue, Playwright (e2e), Node (validation scripts), Crowdin.

## Global Constraints

- Selectable UI languages after this work: **exactly** `en` and `pt-BR`. No other locale files may remain in `frontend/src/i18n/locales/`.
- Locale code / filename for Portuguese: **`pt-BR`** (`pt-BR.json`), native label **`Português (Brasil)`**.
- **Default** active locale for a new user (no saved preference): `pt-BR`. English must still be auto-selected when the browser language base is `en`.
- `pt-BR.json` and `en.json` MUST have identical key sets at all times (enforced by the key-parity script).
- Source/authoring locale stays `en`. `fallbackLocale` stays `en` (source-complete; missing keys degrade to English, never blank — see Task 2 note).
- **Do NOT translate:** database content, WhatsApp templates/messages, user-stored operational content, Go backend strings, or `components/ui/*` shadcn primitives.
- Preserve all interpolation placeholders (`{name}`, `{count}`, …) and ICU plural syntax verbatim when translating.
- Commit after each task. Follow existing key convention `section.subsection.action`.

- [ ] **Step 1: Write the key-parity script**

Create `frontend/scripts/i18n-check-keys.mjs`:

```js
import { readFileSync } from 'node:fs'
import { fileURLToPath } from 'node:url'
import { dirname, join } from 'node:path'

const here = dirname(fileURLToPath(import.meta.url))
const localesDir = join(here, '..', 'src', 'i18n', 'locales')

function flatten(obj, prefix = '', out = new Set()) {
  for (const [k, v] of Object.entries(obj)) {
    const key = prefix ? `${prefix}.${k}` : k
    if (v && typeof v === 'object' && !Array.isArray(v)) flatten(v, key, out)
    else out.add(key)
  }
  return out
}

const en = flatten(JSON.parse(readFileSync(join(localesDir, 'en.json'), 'utf8')))
const pt = flatten(JSON.parse(readFileSync(join(localesDir, 'pt-BR.json'), 'utf8')))

const missingInPt = [...en].filter(k => !pt.has(k)).sort()
const extraInPt = [...pt].filter(k => !en.has(k)).sort()

if (missingInPt.length || extraInPt.length) {
  if (missingInPt.length) {
    console.error(`\n❌ ${missingInPt.length} key(s) missing in pt-BR.json:`)
    missingInPt.forEach(k => console.error('  - ' + k))
  }
  if (extraInPt.length) {
    console.error(`\n❌ ${extraInPt.length} extra key(s) in pt-BR.json (not in en.json):`)
    extraInPt.forEach(k => console.error('  + ' + k))
  }
  process.exit(1)
}
console.log(`✅ en.json and pt-BR.json key sets match (${en.size} keys).`)
```

- [ ] **Step 2: Write the hardcoded-scan script**

Create `frontend/scripts/i18n-scan-hardcoded.mjs`:

```js
import { readFileSync, readdirSync, statSync } from 'node:fs'
import { fileURLToPath } from 'node:url'
import { dirname, join, relative } from 'node:path'

const here = dirname(fileURLToPath(import.meta.url))
const srcDir = join(here, '..', 'src')
const allowlist = JSON.parse(readFileSync(join(here, 'i18n-allowlist.json'), 'utf8'))
const allowSet = new Set(allowlist.strings)
const allowFiles = new Set(allowlist.files)

function walk(dir, files = []) {
  for (const name of readdirSync(dir)) {
    const p = join(dir, name)
    if (statSync(p).isDirectory()) {
      if (name === 'ui') continue // shadcn primitives: no copy
      walk(p, files)
    } else if (name.endsWith('.vue')) {
      files.push(p)
    }
  }
  return files
}

// Text between tags: >Some Text<  (letters, not a binding {{ }})
const textRe = />\s*([A-Za-z][A-Za-z0-9 ,.'!?&/()-]{2,})\s*</g
// Literal attributes: placeholder="Foo" title="Foo" label="Foo" aria-label="Foo"
const attrRe = /\s(placeholder|title|label|aria-label)="([A-Za-z][^"]{2,})"/g

let total = 0
for (const file of walk(srcDir)) {
  const rel = relative(join(here, '..'), file).replace(/\\/g, '/')
  if (allowFiles.has(rel)) continue
  const src = readFileSync(file, 'utf8')
  const hits = []
  let m
  while ((m = textRe.exec(src))) {
    const s = m[1].trim()
    if (s.includes('{{') || allowSet.has(s)) continue
    hits.push(s)
  }
  while ((m = attrRe.exec(src))) {
    if (allowSet.has(m[2])) continue
    hits.push(`${m[1]}="${m[2]}"`)
  }
  if (hits.length) {
    total += hits.length
    console.log(`\n${rel} (${hits.length})`)
    hits.forEach(h => console.log('  • ' + h))
  }
}
if (total) {
  console.error(`\n❌ ${total} hardcoded string(s) found.`)
  process.exit(1)
}
console.log('✅ No hardcoded strings found outside allowlist.')
```

- [ ] **Step 2: Make pt-BR the default with correct Portuguese detection**

Replace the `getDefaultLocale` function body with:

```ts
function getDefaultLocale(): string {
  // Explicit saved preference wins.
  const saved = localStorage.getItem('locale')
  if (saved && messages[saved]) {
    return saved
  }

  // Any Portuguese browser variant (pt, pt-BR, pt-PT...) maps to pt-BR.
  const browserLang = navigator.language.toLowerCase()
  if (browserLang.startsWith('pt') && messages['pt-BR']) {
    return 'pt-BR'
  }

  // Otherwise honor an exact base-language match (e.g. en-US -> en).
  const base = browserLang.split('-')[0]
  if (messages[base]) {
    return base
  }

  // Default for everyone else.
  return messages['pt-BR'] ? 'pt-BR' : 'en'
}
```

NOTE (deliberate refinement of the spec): `fallbackLocale` stays `'en'` — English is the source-complete locale, so any missing key degrades to visible English rather than to a possibly-incomplete pt-BR or a blank. pt-BR is still the *default active* locale, which is what "pt-BR padrão" means. Leave the `createI18n({ ..., fallbackLocale: 'en', ... })` line unchanged.

## Tasks 5–9: Extract hardcoded strings (gated by `npm run i18n:scan`)

Each of these tasks targets a group of files from the Task 1 baseline. The **procedure is identical** for every file; the objective test is the scanner returning zero for that group.

### Extraction procedure (apply per file)

For each hardcoded string in the file:

1. Choose a key under the matching existing section (`chatbot.*`, `calling.*`, `flowBuilder.*`, `flows.*`, `common.*`, `validation.*`, …), name `section.subsection.action`.
2. Add the key to **both** `en.json` (English value) and `pt-BR.json` (Portuguese value).
3. Replace the literal in the component:
   - Template text/attribute → `{{ $t('key') }}` or `:attr="$t('key')"`.
   - Script literal → add `import { useI18n } from 'vue-i18n'` + `const { t } = useI18n()` (once per file), then `t('key')`.
4. If a literal is genuinely non-translatable (icon glyph, brand name, `—`, single symbol), add it to `strings` in `i18n-allowlist.json` instead — do not invent a key.

### Worked example (real code — `components/chatbot/nodes/ChatbotButtonsNode.vue`)

Before (`<template>` tail):
```html
    <p v-else class="text-muted-foreground italic">No buttons</p>
```
The `data?.label || 'Buttons'` default and `b.title || '—'` also contain literals: `'Buttons'` is a fallback node label, `'—'` is an em-dash symbol.

After — add to `en.json` `"chatbot"` section: `"nodes": { "buttons": "Buttons", "noButtons": "No buttons" }`, and the pt-BR equivalents `"Botões"` / `"Sem botões"`. Then:
